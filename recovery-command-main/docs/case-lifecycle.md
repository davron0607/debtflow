# Жизненный цикл дела (Case Lifecycle) — дизайн под право Узбекистана

Статус: **проект на согласование**. Ничего из раздела 4 (целевая модель) ещё не
внедрено в код — это спецификация для следующего этапа. Раздел 5 фиксирует,
что уже сейчас не соответствует праву/логике.

## 1. Зачем этот документ

Текущий `CaseStatus` (`prisma/schema.prisma`) и стейт-машина
(`src/lib/state-machine.ts`) выросли органически и не отражают реальный
юридический процесс взыскания в Узбекистане:

- Поле `enforcementRoute` (`NOTARY` / `COURT`) существует в схеме и
  выставляется через `apiSetRoute`, но **не влияет** ни на один переход в
  `state-machine.ts` — юридически разные маршруты (нотариальная надпись vs
  суд) выглядят в статусах одинаково ("COURT_PACKAGE_READY" звучит как суд,
  даже если выбран нотариус).
- Претензионный порядок (обязателен для экономического суда) — необязательный
  шаг в графе переходов: `ESCALATED_TO_LEGAL` может уйти прямо в
  `COURT_PACKAGE_READY`, минуя `PRE_CLAIM_SENT`.
- Нет статуса "получена исполнительная надпись нотариуса" — только
  судебно-ориентированные названия.
- "МИБ" в интерфейсе — устоявшийся у пользователей термин, официально это
  **Бюро принудительного исполнения (БПИ)** при Генеральной прокуратуре. Нужно
  явное сопоставление терминов, а не переименование UI вслепую.
- Нет SLA-таймеров под реальные процессуальные сроки (претензия, суд, БПИ) —
  есть только `PROMISE_DUE`.

## 3. Два маршрута принудительного взыскания — ключевая развилка

Сейчас граф переходов один для всех дел. По праву РУз маршрут расходится
сразу после `ESCALATED_TO_LEGAL` в зависимости от `enforcementRoute`:

```
                         ┌─────────────────────┐
                         │  ESCALATED_TO_LEGAL  │
                         └──────────┬───────────┘
                                    │
                     обязательна претензия (ст.189 ХПК)
                                    │
                         ┌──────────▼───────────┐
                         │    PRE_CLAIM_SENT     │  SLA: ответ 30 дней
                         └──────────┬───────────┘
                    ┌───────────────┴────────────────┐
        NOTARY (бесспорный долг,                ИЛИ  COURT (спор /
        обычно обеспеченный)                          нотариус недоступен)
                    │                                          │
        ┌───────────▼────────────┐              ┌──────────────▼──────────────┐
        │   NOTARY_INSCRIPTION_  │              │  приказное / упрощённое /    │
        │       REQUESTED        │              │  исковое — выбирается по      │
        └───────────┬────────────┘              │  сумме и наличию спора        │
                     │                            └──────────────┬──────────────┘
        ┌────────────▼────────────┐                              │
        │ NOTARY_INSCRIPTION_     │               ...court sub-flow (см. §4)...
        │      ISSUED             │
        └────────────┬────────────┘
                      │
                      └──────────────┐
                                     ▼
                         ┌───────────────────────┐
                         │   READY_FOR_ENFORCEMENT │   (после суда ИЛИ после нотариуса)
                         │        (БПИ)            │
                         └───────────┬─────────────┘
                                     │
                         ┌───────────▼─────────────┐
                         │  ENFORCEMENT_IN_PROGRESS │  SLA: 5 раб.дн. добровольно
                         │         (БПИ)            │        + до 2 мес. принуд.
                         └───────────┬─────────────┘
                                     │
                        ┌────────────┴────────────┐
                        ▼                          ▼
                    PAID / CLOSED              WRITTEN_OFF
```

Оплата (`PAID`/`PARTIALLY_PAID`) может произойти на **любом** шаге до этого —
не только в конце. Это уже верно учтено в текущей `spineStage()`
(`"resolved"` — отдельная ветка, не привязанная к судебной стадии); эту идею
нужно сохранить и в целевой модели.

## 4. Целевая модель статусов

| Статус (код) | Значение | Условие входа | Условие выхода / SLA | Кто может менять |
|---|---|---|---|---|
| `NEW` | Дело загружено в портфель банка | Импорт портфеля | Назначение агентству/юристу | `BANK_ADMIN` |
| `ASSIGNED` | Назначено агентству/юристу/in-house | `apiAssign` | Старт софт-взыскания или прямая эскалация | `BANK_ADMIN` |
| `SOFT_COLLECTION` | Досудебная работа (звонки/SMS/визиты) | из `ASSIGNED`/`NO_CONTACT`/`PROMISE_BROKEN` | Контакт/обещание/спор/эскалация по DPD-политике банка (не жёсткая юр. норма — см. §2 про 90 дней) | коллектор |
| `CONTACTED` | Контакт состоялся | лог контакта | Обещание / частичная оплата / спор | коллектор |
| `NO_CONTACT` | Должник недоступен | лог контакта | Возврат в софт-взыскание / эскалация после N попыток | коллектор |
| `PROMISE_TO_PAY` | Обещание оплаты зафиксировано | `apiLogPromise` | Оплата / нарушение обещания (SLA = `promisedDate`) | коллектор/бухгалтер |
| `PROMISE_BROKEN` | Обещание не исполнено | SLA `PROMISE_DUE` истёк | Продолжить софт / эскалация | коллектор |
| `PARTIALLY_PAID` | Частичная оплата, остаток > 0 | `apiRecordPayment` | Полная оплата / обещание нарушено | коллектор/бухгалтер |
| `DISPUTE` | Должник оспаривает долг | лог контакта / коллектор | **Только COURT-маршрут** — нотариальная надпись невозможна для спорного долга | `BANK_ADMIN`/`BANK_LEGAL` |
| `RESTRUCTURING_PROPOSED` | Банк предложил реструктуризацию | из `DISPUTE` | Утверждена / отказ → эскалация | `BANK_ADMIN`/`BANK_LEGAL` |
| `RESTRUCTURED` | Новый график согласован | утверждение | Закрытие по новому графику | `BANK_ADMIN`/`BANK_LEGAL` |
| `ESCALATED_TO_LEGAL` | Передано в юр. отдел/юрфирму | эскалация из любой pre-стадии | **Обязательна** претензия (ст. 189 ХПК) — не пропускать | `BANK_ADMIN` |
| `PRE_CLAIM_SENT` | Претензия направлена | из `ESCALATED_TO_LEGAL` | SLA 30 дней на ответ; далее развилка NOTARY/COURT | `BANK_LEGAL`/`LEGAL_FIRM` |
| `NOTARY_INSCRIPTION_REQUESTED` *(новый)* | Подан пакет нотариусу | `enforcementRoute=NOTARY`, долг бесспорный | Надпись выдана / нотариус отказал (спор → в суд) | `BANK_LEGAL`/`LEGAL_FIRM` |
| `NOTARY_INSCRIPTION_ISSUED` *(новый)* | Исполнительная надпись получена | нотариус выдал | → передача в БПИ | `BANK_LEGAL`/`LEGAL_FIRM` |
| `COURT_PACKAGE_READY` | Пакет для суда готов | `enforcementRoute=COURT` | Подача (приказное/упрощённое/исковое — по сумме/спору) | `BANK_LEGAL`/`LEGAL_FIRM` |
| `FILED_TO_COURT` | Иск/заявление подано | подача | Решение / приказ / переход в исковое при возражении (SLA см. §2) | `BANK_LEGAL`/`LEGAL_FIRM` |
| `COURT_DECISION_RECEIVED` | Решение вступило в силу | суд | → передача в БПИ (SLA предъявления — 3 года) | `BANK_LEGAL`/`LEGAL_FIRM` |
| `READY_FOR_ENFORCEMENT` *(переименовать `READY_FOR_MIB`)* | Исполнительный документ готов к предъявлению в БПИ | решение суда ИЛИ надпись нотариуса | Предъявление в БПИ | `BANK_ADMIN`/`BANK_LEGAL` |
| `ENFORCEMENT_IN_PROGRESS` *(новый)* | Дело в БПИ | предъявлено | SLA: 3 раб.дня на возбуждение, 5 раб.дней добровольно, до 2 мес. принудительно | `BANK_ADMIN`/`BANK_LEGAL` |
| `PAID` | Полностью оплачено | остаток = 0 | Закрытие | коллектор/бухгалтер (см. фикс от 07.07 — с проверкой остатка) |
| `CLOSED` | Дело закрыто | после `PAID`/`RESTRUCTURED` | терминальный | `BANK_ADMIN` |
| `WRITTEN_OFF` | Списано | из `ENFORCEMENT_IN_PROGRESS` при невозможности взыскания | терминальный | `BANK_ADMIN` |

Статусы, помеченные *(новый)*/*(переименовать)* — предложение, требует
отдельной миграции и согласования (не проект на сегодня).

## 5. Конкретные несоответствия в текущем коде (`state-machine.ts`, `api.ts`)

1. **`enforcementRoute` не используется в переходах.** `apiSetRoute`
   (`src/lib/api.ts:764`) пишет `NOTARY`/`COURT` в БД, но `state-machine.ts`
   ведёт по одному и тому же графу `PRE_CLAIM_SENT → COURT_PACKAGE_READY →
   FILED_TO_COURT → COURT_DECISION_RECEIVED` независимо от выбора. Из-за
   этого статус "COURT_PACKAGE_READY" показывается и когда реально идёт
   нотариус — отсюда ощущение хаоса.
2. **Претензия — необязательный шаг.** `ESCALATED_TO_LEGAL` в
   `state-machine.ts:104-107` разрешает переход сразу в
   `COURT_PACKAGE_READY`, минуя `PRE_CLAIM_SENT`, хотя претензия обязательна
   для экономического суда (ст. 189 ХПК).
3. **`READY_FOR_MIB` — понятное пользователям, но не официальное название.**
   Официально — Бюро принудительного исполнения (БПИ) при Генпрокуратуре.
   Не переименовывать в UI без решения продукта, но зафиксировать
   сопоставление в документации/коде (комментарий).
4. **Нет SLA-таймеров под юридические сроки.** Модель `SlaTimer` (`type`,
   `dueAt`, `breached`) сейчас создаётся только для `PROMISE_DUE`
   (`apiLogPromise`). Нет таймеров на: ответ на претензию (30 дней),
   возбуждение в БПИ (3 раб.дня), добровольное исполнение (5 раб.дней),
   принудительное исполнение (2 месяца), срок предъявления исполнительного
   документа (3 года).
5. **DPD-эскалация — без явной политики.** Нет автоматического/рекомендованного
   перехода `SOFT_COLLECTION → ESCALATED_TO_LEGAL` по DPD-порогу. Ставить
   ли 90 дней как в Указе УП-1154 — вопрос политики банка/продукта, не жёсткая
   юридическая норма для розничных банковских кредитов (это общее правило
   для дебиторской задолженности предприятий, не банк-специфичный NPL-класс
   ЦБ) — предлагаю сделать **настраиваемым полем на уровне организации**
   (`Organization.escalationDpdThreshold`), а не хардкодить.
6. **`DISPUTE` не блокирует NOTARY-маршрут.** Нотариальная надпись требует
   бесспорности долга (см. §2). Сейчас ничего не мешает выбрать
   `enforcementRoute=NOTARY` для дела в статусе `DISPUTE`.
7. **[Исправлено 10.07] Оплата "терялась" из статусов без прямого ребра к
   `PAID`.** Найдено первым прогоном `case-lifecycle-qa`-агента:
   `apiRecordPayment` определял целевой статус по фактическому остатку
   (верно), но применял его только если `canTransition(c.status, to, role)`
   разрешал такой переход по графу `T`. У статусов `SOFT_COLLECTION`,
   `NO_CONTACT`, `PROMISE_BROKEN`, `DISPUTE`, `ASSIGNED` нет прямого ребра к
   `PAID`/`PARTIALLY_PAID` — деньги списывались, остаток обнулялся, но
   статус и воронка оставались на досудебной стадии. Это прямое нарушение
   принципа §3 ("оплата может произойти на любом шаге"). Фикс: оплата — факт,
   а не переход по графу; статус теперь обновляется напрямую при получении
   денег (кроме уже терминальных `CLOSED`/`WRITTEN_OFF`), без обращения к
   `canTransition`.

## 6. Что делать дальше

Это документ для согласования, а не готовый PR. Рекомендуемый порядок
внедрения (по одному пункту за раз, с проверкой на реальных делах):

1. Завести `courtTrack`-подобное явное отражение `enforcementRoute` в
   лейблах переходов (без смены `CaseStatus` enum) — самый дешёвый и
   безопасный первый шаг.
2. Сделать `PRE_CLAIM_SENT` обязательным перед `COURT_PACKAGE_READY`.
3. Заблокировать `apiSetRoute(NOTARY)` для дел в статусе `DISPUTE`.
4. Добавить SLA-таймеры под юридические сроки из §2.
5. Только затем — миграция `READY_FOR_MIB → READY_FOR_ENFORCEMENT` +
   `ENFORCEMENT_IN_PROGRESS` + нотариальные статусы (структурное изменение
   схемы, нужна отдельная миграция и ревью).

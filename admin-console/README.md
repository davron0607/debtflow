# DebtFlow · Консоль оператора платформы

Отдельное приложение — сознательно НЕ тот же фронтенд, что у банков и агентств
(`recovery-command-main`). Причина: общий фронтенд с админ-правами означает
общий blast radius при компрометации, общие окна обслуживания и невозможность
независимо откатывать/гасить одно без другого. Здесь — свой процесс, свой
Railway-сервис, свой (под)домен, своя cookie, и ограниченная роль Postgres —
даже баг в этом коде физически не даст прочитать `Case`/`Payment`/`Document`
банков и агентств.

## Локальный запуск

```bash
npm install
cp .env.example .env   # заполнить DATABASE_URL ограниченной ролью
npm run dev
```

Роль и права для `DATABASE_URL` — см. `../scripts/create-admin-console-role.sql`
(выполняется один раз от имени владельца БД).

## Бутстрап первого оператора

`prisma/ensure-ops.ts` создаёт `Organization(PLATFORM)` и первого
`User(PLATFORM_ADMIN)` — это INSERT, которого у ограниченной runtime-роли
**нет**. Запускать с полными правами (DATABASE_URL владельца, не runtime-роли
сервиса):

```bash
DATABASE_URL=<owner-url> PLATFORM_ADMIN_PASSWORD=<strong-pw> npm run db:ensure-ops
```

## Деплой на Railway

1. New Service → Deploy from GitHub repo, **Root Directory = `admin-console`**.
2. Переменные: `DATABASE_URL` (ограниченная роль), `RESEND_API_KEY`, `MAIL_FROM`.
3. Домен: отдельный поддомен, например `admin.debtflow.uz`.
4. Один раз — бутстрап оператора (см. выше) через Railway Console с owner-URL.

## Что не входит в это приложение

- Смена пароля оператора — через прямой доступ к БД (owner creds), не веб-форма.
- Регистрация/подтверждение e-mail для `PLATFORM_ADMIN` — этот аккаунт создаётся
  только бутстрап-скриптом, самостоятельная регистрация недоступна.
- Доступ к делам, платежам, документам банков/агентств — не реализовано и не
  должно быть реализовано (граница нейтралитета платформы).
